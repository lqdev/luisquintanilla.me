## Discovery

IndieWeb microformat compliance requires specific HTML markup (like `u-author h-card` with `u-photo`, `u-url`, and `p-name` properties) to be present in every `h-entry`. However, displaying this author information on every post creates visual redundancy on a personal site where the author is always the same person. The tension between machine-readable requirements and human-readable design seemed like it required a compromise — until we discovered that strategic CSS hiding resolves it completely.

## Root Cause / Problem

The IndieWeb ecosystem relies on microformats2 markup for interoperability. Webmention endpoints, feed readers (like Aperture, Microsub readers), and IndieWeb tools parse HTML to find author information, reply contexts, and content structure. Without proper `h-card` markup embedded in each `h-entry`, these tools cannot attribute content correctly.

On a personal website, though, every post is by the same author. Displaying "by Luis Quintanilla" with an avatar photo on every single post, note, and response adds visual clutter without providing useful information to human readers. Removing the markup breaks IndieWeb compatibility; keeping it visible degrades the design.

## Solution

Include the complete microformat markup in the HTML but hide it visually using a dedicated CSS class:

**HTML Structure** — Full `h-card` with all required IndieWeb properties:

```html
<div class="u-author h-card microformat-hidden">
  <img class="u-photo" src="/avatar.jpg" alt="Author avatar" />
  <a class="u-url p-name" href="/">Luis Quintanilla</a>
</div>
```

**CSS Hiding Strategy** — A purpose-built class that hides content from visual rendering while keeping it accessible to parsers:

```css
.microformat-hidden {
  display: none !important;
  visibility: hidden !important;
}
```

The `!important` declarations ensure the hiding is not accidentally overridden by other styles. Both `display: none` and `visibility: hidden` are used together for maximum compatibility — some parsers respect one but not the other.

**Parser Compatibility** — Webmention parsers, feed readers, and IndieWeb validation tools parse the raw HTML document, not the visually rendered page. They find the `h-card` markup regardless of CSS visibility. This approach is explicitly supported by the IndieWeb community as a valid technique for including machine-readable metadata.

## Key Components

- **`u-author h-card`**: The microformats2 class combination that identifies an author card within an entry
- **`u-photo`**: Author avatar image, required by many IndieWeb consumers for proper attribution display
- **`u-url`**: Author's canonical URL, used for identity verification in Webmention flows
- **`p-name`**: Author's display name, used in feed readers and reply contexts
- **`microformat-hidden`**: The project-specific CSS class that hides the markup from visual rendering
- **ViewEngine rendering**: The `h-card` markup is generated by Giraffe ViewEngine as part of each entry's render function

## Results

- Full IndieWeb microformat compliance verified by indiewebify.me validation tools
- Clean visual design with no redundant author attribution on individual entries
- Webmention sending and receiving works correctly with proper author identification
- Feed readers display author information pulled from the hidden `h-card` markup
- No JavaScript required — the solution is pure HTML and CSS

## Benefits

This pattern resolves the tension between semantic web standards and visual design without compromise. IndieWeb tools get the structured data they need, and human readers get a clean interface. The approach is also low-maintenance — the hidden `h-card` is generated once in the shared entry rendering function and automatically included in every content type. It requires no per-entry configuration and no JavaScript runtime. As a bonus, the pattern serves as a useful example of the broader principle that machine-readable and human-readable requirements can coexist through separation of concerns in the presentation layer.
