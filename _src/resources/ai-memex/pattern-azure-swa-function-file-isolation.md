# Pattern: Azure SWA Managed Functions Cannot Access Static Content Files

## Discovery

Built a SPARQL endpoint as an Azure SWA managed function (`api/function_app.py`) that needed to load Turtle (`.ttl`) RDF graph files at startup. The graph files were generated by CI into `graph/articles/` and the function tried to read them via `Path(__file__).parent.parent / "graph" / "articles"`.

The endpoint returned valid SPARQL JSON responses — but with zero results. No errors, no crashes, just empty bindings. The function was running fine; it simply couldn't find any files to load.

## Root Cause

Azure Static Web Apps deploys **two isolated environments**:

1. **Static content** (`app_location`): Served via CDN — HTML, CSS, JS, JSON, etc.
2. **Managed Functions** (`api_location`): Runs in a separate Azure Functions sandbox

These environments **do not share a filesystem**. The function can only see files within its own `api_location` directory tree. Any path traversal outside (`../graph/`, `../../content/`) resolves to nothing — `Path.exists()` returns `False` silently.

This is different from local development where everything is in the same repo directory and relative paths "just work."

## Solution

Copy the data files into the API directory during CI, so they're deployed as part of the function package:

```yaml
# In the KG build workflow, after generating graph artifacts:
- name: Copy graph data into API package
  run: |
    mkdir -p api/data
    cp graph/articles/*.ttl api/data/ 2>/dev/null || echo "No TTL files to copy"

- name: Commit graph artifacts
  run: |
    git add graph/ api/data/
    git commit -m "chore: update knowledge graph [skip ci]"
    git push
```

And update the function to read from its own directory:

```python
# Before (broken in SWA):
graph_dir = Path(__file__).parent.parent / "graph" / "articles"

# After (works in SWA):
graph_dir = Path(__file__).parent / "data"
```

## Prevention

When designing Azure SWA architectures with managed functions:

- **Treat `app_location` and `api_location` as completely separate deployments** — they share nothing at runtime
- **If a function needs data files, put them inside `api/`** — either checked in or copied during CI
- **Test with `swa start` locally** — the SWA CLI emulator reproduces this isolation, unlike raw `func start`
- **Watch for "silent empty"** — the function won't crash if the directory doesn't exist; it just loads zero data, which surfaces as empty query results rather than errors
